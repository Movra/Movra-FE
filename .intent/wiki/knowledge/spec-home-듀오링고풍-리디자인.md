# Spec: HOME 듀오링고풍 리디자인

> status: 🚧 draft (awaiting human approval — review then `intent spec approve <slug>`)

## 문제 정의

HOME(`src/features/core-loop/CoreLoopDashboard.tsx`)이 '복잡하고 확 와닿지 않고 AI스럽다'. 진단 결과 원인은 디자인 감각 부재가 아니라 **의도와 구현의 간극**:

- `docs/design.md`에 강한 디자인 의도(새싹 캐릭터·유기적 형태·green-first·차분 에디토리얼)와 토큰(`tokens.css`), 캐릭터 자산 6종이 이미 존재한다.
- 그러나 HOME은 한 화면에 7개 섹션(TopPick·회복·시간표·목표 시각화·시험·Focus·요약 4지표)을 균일 카드로 동시에 펼쳐 위계가 없다 → '균일 카드 그리드 + 정보 과밀 + 개성 없는 색/타이포'.
- 캐릭터는 각 카드 구석에 장식으로 흩뿌려져 'Duolingo풍의 캐릭터 주인공'이 되지 못한다.

## 목표

HOME 한 화면을 'Duolingo풍 분위기'로 다시 잡는다: **큰 캐릭터 주인공 + 한 화면 한 가지 행동(TopPick → 지금 시작) + 따뜻한 격려 말투 + 부드러운 진행 피드백**. 단 PRD가 금지하는 중독성 게임화 메커니즘은 배제한다. 범위는 우선 HOME 1페이지(핵심 1~2 페이지 집중).

## 가정 (확정)

- '느낌'의 지향은 Duolingo의 **분위기·캐릭터·격려 톤**이며, 게임화 **메커니즘이 아니다**.
- 첫 화면(스크롤 전)에는 **캐릭터 + TopPick + 지금 시작** 만 둔다. 나머지 섹션(시간표·목표 시각화·시험·요약)은 **스크롤 아래로 접거나 요약 칩으로 강등**한다.
- 캐릭터(`character-*.webp`)를 장식이 아니라 화면의 주인공으로 승격한다.
- green-first 브랜드 색은 유지한다 (색을 확 바꾸지 않는다).
- API 호출·응답 타입·쿼리 키 등 데이터 구조는 그대로 둔다. 순수 프론트엔드 시각/레이아웃/카피만 수정한다.
- 기능 동작·라우팅, 그리고 페이지 테스트(.test.tsx)는 '절대 불가'로 지정되지 않았다 → 필요 시 수정 허용하되, CLAUDE.md 원칙대로 변경은 최소·외과적으로 한다.

## 명시적 비목표 (PRD 안티비전·안전선과 정합)

- ❌ 누적 스트릭(🔥) — '끊기면 손해'는 손실 프레이밍 → PRD §4-3 편도체 위협 회로 회피 위반.
- ❌ 포인트/XP/뱃지(⭐) 남발 — 변동 보상 도파민 중독 회피.
- ❌ 랭킹·친구 대비 비교 — 사회적 비교 우울 안전선.
- ❌ 손실/죄책감 어휘 ('어제 0시간', '놓쳤어요') — Recovery 원칙 위반.
- ❌ 색 시스템 전면 교체, API/데이터 구조 변경, 전면(전 페이지) 리디자인 — 이번 범위 아님.

## 경계 예시 (되는 것 / 안 되는 것)

- ✅ 큰 캐릭터가 '오늘은 이것만 하면 돼!' 격려 → 안 됨: 캐릭터가 '연속 3일째예요, 깨지마요!' 압박.
- ✅ 오늘 TopPick 진행 노드(●━━○)·달성률 표시 → 안 됨: 며칠 연속 스트릭 카운터.
- ✅ Tiny Win을 '오늘 +2'처럼 그날 축하 → 안 됨: 누적 포인트 잔액 ⭐120.
- ✅ 첫 화면은 행동 1개, 시간표/시험/요약은 스크롤 아래 → 안 됨: 7개 패널 동시 노출.
- ✅ green 브랜드 안에서 위계·여백·타이포로 개성 → 안 됨: 색을 보라/파랑 그라데이션으로 교체.

## 성공 기준 (눈으로 확인)

1. HOME 첫 화면(스크롤 전)에 캐릭터 + TopPick 한 줄 + '지금 시작' 버튼만 보인다 — 패널 7개 동시 노출 사라짐. → 검증: 데스크톱/모바일 첫 뷰포트 스크린샷.
2. 캐릭터가 화면의 시각적 주인공 크기로 등장한다 (구석 장식 아님).
3. 스트릭·포인트·랭킹·손실 어휘가 어디에도 없다. → 검증: 코드/카피 grep.
4. 시간표·목표·시험·요약은 스크롤 아래 또는 요약 칩으로 강등됐다.
5. green-first 토큰만 사용, API/쿼리 키/응답 타입 변경 0건. → 검증: `git diff` 범위 확인.
6. 빌드·타입체크 통과. (테스트는 레이아웃 변경에 맞춰 필요 시 갱신)
