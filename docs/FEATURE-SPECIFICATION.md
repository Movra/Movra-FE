# Movra 기능 명세서

## 0. 문서 목적

이 문서는 Movra 백엔드가 제공해야 하는 기능을 사용자 관점과 시스템 관점에서 정리한다.

API 세부 요청/응답은 `API-specification.md`가 담당하고, 이 문서는 다음을 명확히 한다.

- 기능의 목적
- 주요 사용자 흐름
- 기능별 정책과 제약
- 백엔드 처리 기준
- 수용 기준
- 현재 구현 상태

---

## 1. 제품 기능 범위

Movra는 고등학생의 하루 학습 실행을 돕는 생산성 백엔드다. 핵심 루프는 다음 순서로 동작한다.

1. 온보딩에서 학습 성향과 시험 정보를 설정한다.
2. Future Vision으로 장기 목표를 입력한다.
3. MindSweep으로 머릿속 할 일을 꺼낸다.
4. TopPick으로 오늘 반드시 지킬 핵심 할 일을 고른다.
5. Timetable로 오늘의 시간 블록을 잡는다.
6. Focus Session으로 짧게 시작하고 기록한다.
7. Tiny Win과 Daily Reflection으로 실행 직후 보상과 회고를 남긴다.
8. Recovery Card로 실패 다음 날 복귀를 돕는다.
9. Notification, Accountability, Statistics, Home Today가 루프 지속을 보조한다.

## 3. 공통 정책

### 3-1. 인증

- JWT Bearer 토큰 기반 인증을 사용한다.
- 비인증 허용 엔드포인트를 제외한 모든 API는 인증이 필요하다.
- 현재 사용자는 `CurrentUserQuery.currentUser().userId()`로 조회한다.
- Access Token과 Refresh Token은 용도를 구분해야 한다.
- Refresh Token은 원문 저장 없이 해시된 값으로 저장한다.

**수용 기준**

- 인증이 필요한 API는 토큰 없이 호출할 수 없다.
- 만료되거나 잘못된 JWT는 공통 에러 응답으로 처리된다.
- Access Token으로 Refresh Token 재발급을 할 수 없다.
- Refresh Token으로 일반 인증을 할 수 없다.

### 3-2. 에러 처리

- 비즈니스 예외는 `CustomException(ErrorCode)` 패턴을 따른다.
- 도메인 규칙 위반은 `domain/exception`에 둔다.
- 조회 실패 또는 유스케이스 실패는 `application/exception`에 둔다.
- API 에러 응답은 `ErrorResponse` 형식으로 통일한다.

**수용 기준**

- 클라이언트는 모든 비즈니스 실패에서 일관된 `status`, `message`를 받는다.
- 예상하지 못한 예외는 내부 서버 오류로 변환된다.

### 3-3. 시간

- 서비스 로직에서 날짜/시간 정책이 필요한 경우 `Clock` 주입을 우선 사용한다.
- 한국 학생 대상 서비스이므로 기본 시간대는 `Asia/Seoul`이다.
- 날짜 기능은 서버 시간대 기준으로 일관되어야 한다.

**주의**

- 일부 기존 코드에는 `LocalDate.now()`, `LocalDateTime.now()`, `Instant.now()` 직접 호출이 남아 있다.
- 테스트 가능성과 시간대 일관성을 위해 `Clock` 기반으로 점진 정리하는 것이 좋다.

## 4. 인증 및 사용자

### 4-1. 로컬 회원가입

사용자는 이메일, 계정 ID, 프로필명, 프로필 이미지, 비밀번호로 가입한다.

**기능 요구사항**

- 계정 ID는 중복될 수 없다.
- 이메일은 중복될 수 없다.
- 프로필 이미지는 필수다.
- 비밀번호는 해시 후 저장한다.
- 가입 성공 시 `SIGNUP` 분석 이벤트를 기록한다.
- 가입 중 DB 중복이 발생하면 업로드된 이미지를 정리한다.

**수용 기준**

- 중복 계정 ID는 `DUPLICATE_ACCOUNT_ID`로 실패한다.
- 중복 이메일은 `DUPLICATE_EMAIL`로 실패한다.
- 저장 실패 시 업로드된 프로필 이미지가 정리된다.

### 4-2. 로그인

사용자는 계정 ID와 비밀번호로 로그인한다.

**기능 요구사항**

- 계정 ID로 사용자를 조회한다.
- 비밀번호 해시를 비교한다.
- 성공 시 Access Token과 Refresh Token을 발급한다.
- Refresh Token은 해시해서 저장한다.

**수용 기준**

- 존재하지 않는 계정 ID는 로그인할 수 없다.
- 비밀번호가 다르면 로그인할 수 없다.
- 로그인 성공 응답에는 두 토큰이 모두 포함된다.

### 4-3. OAuth 프로필 설정

OAuth 인증 후 pending token을 가진 사용자는 서비스 프로필을 완성한다.

**기능 요구사항**

- pending token으로 OAuth 이메일과 provider를 조회한다.
- 계정 ID 중복을 검사한다.
- 동일 이메일/provider 조합 중복을 검사한다.
- 프로필 이미지와 비밀번호를 포함해 User를 생성한다.
- 가입 성공 시 JWT를 발급하고 pending token을 제거한다.
- 가입 성공 시 `SIGNUP` 분석 이벤트를 기록한다.

**수용 기준**

- 유효하지 않은 pending token은 실패한다.
- 이미 가입된 OAuth 사용자는 중복 가입할 수 없다.
- 프로필 설정 완료 후 즉시 로그인 가능한 토큰이 발급된다.

### 4-4. 온보딩 컨텍스트

클라이언트는 가입/온보딩 화면에서 현재 학교 시간 여부를 조회할 수 있다.

**기능 요구사항**

- 현재 시간이 학교 시간대인지 반환한다.
- 인증 없이 호출 가능하다.
- 학교 시간 판단은 `SchoolHoursPolicy`를 사용한다.

**수용 기준**

- 학교 시간대에는 `pendingSchoolHours=true`를 반환한다.
- 학교 시간대가 아니면 `pendingSchoolHours=false`를 반환한다.

## 5. 개인화와 시험 일정

### 5-1. 행동 프로필

행동 프로필은 사용자의 실행 난이도, 사회적 선호, 회복 스타일, 시험 트랙, 집중 선호 시간, 코칭 모드를 저장한다.

**기능 요구사항**

- 사용자당 하나의 행동 프로필만 생성할 수 있다.
- 필수 응답 값이 누락되면 생성/수정할 수 없다.
- 선호 집중 시작 시간은 종료 시간보다 빨라야 한다.
- 실행 난이도는 TopPick 개수 제한에 영향을 준다.
- 회복 스타일과 코칭 모드는 Recovery Card 메시지에 영향을 준다.

**수용 기준**

- 이미 프로필이 있으면 중복 생성할 수 없다.
- 내 프로필만 조회/수정할 수 있다.
- 유효하지 않은 시간 범위는 실패한다.

### 5-2. 시험 일정

사용자는 내신, 모의고사, 수능 등 시험 일정을 관리한다.

**기능 요구사항**

- 시험명, 시험 유형, 시험 날짜, 과목을 저장한다.
- 사용자별 시험 일정만 조회한다.
- 다음 시험 일정을 조회할 수 있다.
- 시험까지 남은 일수를 계산한다.
- 시험 유형과 남은 일수로 시즌 모드를 계산한다.

**수용 기준**

- 지난 시험은 다음 시험으로 반환하지 않는다.
- 시험 날짜가 가까워지면 시즌 모드가 강화된다.
- 다른 사용자의 시험 일정은 조회/수정/삭제할 수 없다.

## 6. Future Vision

Future Vision은 사용자의 장기 목표와 이번 주 목표를 저장한다.

**기능 요구사항**

- 사용자당 하나의 Future Vision만 생성할 수 있다.
- 연간 비전과 주간 비전은 필수다.
- 연간 비전 수정 시 생성/갱신 날짜를 관리한다.
- 주간 비전은 별도로 수정할 수 있다.

**수용 기준**

- 이미 Future Vision이 있으면 중복 생성할 수 없다.
- 비어 있거나 길이 제한을 넘는 비전은 저장할 수 없다.
- 내 Future Vision만 조회/수정할 수 있다.

## 7. Daily Plan

Daily Plan은 하루 계획의 루트다. MindSweep, MorningTask, TopPick은 DailyPlan 안의 Task로 관리된다.

### 7-1. DailyPlan 생성/조회

**기능 요구사항**

- 사용자와 날짜 조합은 유일하다.
- 오늘 계획이 없으면 `findOrCreateToday`로 자동 생성할 수 있다.
- DailyPlan 생성 시 Timetable이 도메인 이벤트로 자동 생성된다.

**수용 기준**

- 같은 날짜에 같은 사용자의 DailyPlan이 2개 생기면 안 된다.
- 동시 요청으로 중복 생성이 발생해도 중복 키 예외를 처리해야 한다.
- DailyPlan 생성 후 연결된 Timetable이 존재해야 한다.

### 7-2. MindSweep

MindSweep은 사용자가 떠오르는 할 일을 자유롭게 입력하는 기능이다.

**기능 요구사항**

- DailyPlan에 일반 Task를 추가한다.
- 일반 Task는 수정, 삭제, 완료, 완료 취소할 수 있다.
- 완료된 Task는 수정/삭제할 수 없다.

**수용 기준**

- 존재하지 않는 DailyPlan에는 Task를 추가할 수 없다.
- 존재하지 않는 Task는 수정/삭제/완료 처리할 수 없다.
- MorningTask와 일반 Task의 타입을 혼동해서 처리할 수 없다.

### 7-3. MorningTask

MorningTask는 다음 날 첫 행동을 돕는 보조 Task다.

**기능 요구사항**

- 특정 날짜의 DailyPlan에 MorningTask를 추가한다.
- MorningTask는 조회, 수정, 삭제, 완료, 완료 취소할 수 있다.
- targetDate 기준으로 조회할 수 있다.

**수용 기준**

- MorningTask API는 Morning Task 타입만 다룬다.
- 일반 MindSweep Task를 MorningTask API로 조작할 수 없다.

### 7-4. TopPick

TopPick은 오늘 반드시 지킬 핵심 Task다.

**기능 요구사항**

- MindSweep Task 중 TopPick을 선택한다.
- TopPick에는 예상 소요 시간과 메모가 포함된다.
- 사용자의 실행 난이도에 따라 최대 선택 개수를 제한한다.
- TopPick 선택/해제 시 Timetable에 도메인 이벤트가 전달된다.
- TopPick 슬롯 시간이 변경되면 예상 소요 시간에 반영된다.

**수용 기준**

- 최대 개수를 초과해 TopPick을 선택할 수 없다.
- TopPick이 아닌 Task의 예상 시간은 수정할 수 없다.
- TopPick 선택 이벤트는 Timetable의 topPickTotal에 반영된다.

## 8. Timetable

Timetable은 하루 시간 블록을 관리한다.

### 8-1. Timetable 생성/조회

**기능 요구사항**

- DailyPlan이 생성되면 Timetable이 자동 생성된다.
- Timetable은 DailyPlan과 1:1로 연결된다.
- DailyPlan ID로 Timetable을 조회할 수 있다.

**수용 기준**

- 존재하지 않는 Timetable 조회는 실패한다.
- 다른 사용자의 Timetable을 조작할 수 없어야 한다.

### 8-2. Slot 배정

**기능 요구사항**

- TopPick Task를 TopPick Slot으로 배정할 수 있다.
- 일반 Task를 Task Slot으로 배정할 수 있다.
- 직접 입력 Slot을 추가할 수 있다.
- Slot은 시작 시간이 종료 시간보다 빨라야 한다.
- 같은 Timetable 안에서 Slot 시간이 겹치면 안 된다.
- TopPick Slot이 모두 배정되기 전에는 일반 Task Slot 배정을 제한한다.

**수용 기준**

- 유효하지 않은 시간 범위는 실패한다.
- 겹치는 시간대는 실패한다.
- TopPick Slot 제한을 초과할 수 없다.

### 8-3. Slot 수정/삭제

**기능 요구사항**

- Slot 시간을 재조정할 수 있다.
- Slot을 삭제할 수 있다.
- TopPick Slot 재조정 시 DailyPlan의 예상 소요 시간이 갱신된다.

**수용 기준**

- 존재하지 않는 Slot은 수정/삭제할 수 없다.
- 재조정 후에도 시간 겹침 검증이 적용된다.

## 9. Focus Session

Focus Session은 짧은 집중 실행을 기록한다.

### 9-1. 집중 시작

**기능 요구사항**

- 사용자는 집중 세션을 시작할 수 있다.
- 허용 프리셋은 3, 5, 10, 25분이다.
- 이미 진행 중인 세션이 있으면 새 세션을 시작할 수 없다.
- 집중 시작 시 `FOCUS_SESSION_STARTED` 분석 이벤트를 기록한다.

**수용 기준**

- 허용되지 않은 프리셋은 실패한다.
- 진행 중 세션이 있으면 `FOCUS_SESSION_ALREADY_IN_PROGRESS`로 실패한다.

### 9-2. 집중 종료

**기능 요구사항**

- 진행 중인 세션을 종료한다.
- 종료 시점은 시작 시점보다 빠를 수 없다.
- durationSeconds를 계산한다.
- 완료 이벤트와 분석 이벤트를 기록한다.

**수용 기준**

- 진행 중 세션이 없으면 종료할 수 없다.
- 이미 완료된 세션은 다시 완료할 수 없다.
- 응답에는 완료 여부, 경과 시간, 프리셋 달성률이 포함된다.

### 9-3. 오늘 집중 조회

**기능 요구사항**

- 오늘의 집중 세션 목록을 조회한다.
- 진행 중인 세션을 포함한다.
- 총 집중 시간을 계산한다.
- 현재 집중 중인지 표시한다.

**수용 기준**

- 오늘 날짜 기준은 서비스 시간대와 일치해야 한다.
- 진행 중 세션의 경과 시간은 조회 시점 기준으로 계산된다.

## 10. Recovery Card

Recovery Card는 실패 후 복귀를 돕는 기능이다.

**기능 요구사항**

- 어제 집중 시간, TopPick 완료율, 최근 공백, 시험 시즌 등을 기준으로 회복 타입을 계산한다.
- 회복 타입은 `NONE`, `NO_FOCUS_YESTERDAY`, `LOW_FOCUS`, `TOP_PICK_MISSED`, `POST_EXAM`, `LONG_ABSENCE` 등을 포함한다.
- 행동 프로필의 회복 스타일과 코칭 모드를 반영해 메시지를 생성한다.
- 추천 집중 시간 또는 다음 행동 힌트를 제공한다.
- 사용자가 회복 카드 액션을 수행하면 `RECOVERY_CARD_ACTIONED` 분석 이벤트를 기록한다.

**수용 기준**

- 회복이 필요 없으면 `needsRecovery=false`를 반환한다.
- 회복이 필요하면 회복 타입과 메시지를 반환한다.
- 액션 기록은 주요 기능 실패를 유발하지 않도록 안전하게 처리되어야 한다.

## 11. Feedback

### 11-1. Tiny Win

Tiny Win은 작은 성취를 저장하는 기능이다.

**기능 요구사항**

- 사용자는 제목과 내용으로 Tiny Win을 생성한다.
- 생성일을 저장한다.
- 내 Tiny Win 목록과 상세를 조회한다.
- 제목과 내용을 각각 수정할 수 있다.
- Tiny Win을 삭제할 수 있다.

**수용 기준**

- 제목과 내용은 비어 있을 수 없다.
- 길이 제한을 초과할 수 없다.
- 다른 사용자의 Tiny Win을 조회/수정/삭제할 수 없다.

### 11-2. Daily Reflection

Daily Reflection은 하루 회고와 If-Then 계획을 저장한다.

**기능 요구사항**

- 특정 날짜의 회고를 생성한다.
- 잘한 점, 무너진 지점, if 조건, then 행동을 저장한다.
- 같은 날짜에 같은 사용자는 하나의 회고만 작성할 수 있다.
- 날짜 기준으로 회고를 조회한다.
- 기존 회고를 수정할 수 있다.

**수용 기준**

- 중복 회고 생성은 실패한다.
- 필수 회고 필드가 비어 있으면 실패한다.
- 내 회고만 조회/수정할 수 있다.

## 12. Study Room

Study Room은 친구와 함께 공부하는 실시간 공간이다.

### 12-1. 방 생성

**기능 요구사항**

- 사용자는 공개방 또는 비공개방을 생성할 수 있다.
- 방 이름은 길이 제한을 가진다.
- 방 생성자는 방장이 된다.
- 비공개방은 초대 코드를 가진다.
- 방 생성 이벤트로 생성자가 Participant에 자동 등록된다.

**수용 기준**

- 방 생성 후 생성자는 참여자 목록에 포함된다.
- 비공개방 상세에는 초대 코드 검증이 적용된다.

### 12-2. 방 목록/상세 조회

**기능 요구사항**

- 공개방 목록을 조회할 수 있다.
- 방 상세는 참여자만 조회할 수 있다.
- 상세 응답에는 참여자 목록과 프로필명이 포함된다.

**수용 기준**

- 참여자가 아닌 사용자는 방 상세를 조회할 수 없다.
- 참여자 프로필명은 User 정보에서 조회된다.

### 12-3. 입장/퇴장/강퇴

**기능 요구사항**

- 공개방은 초대 코드 없이 입장할 수 있다.
- 비공개방은 초대 코드가 필요하다.
- 이미 참여 중인 방에 중복 입장할 수 없다.
- 퇴장 시 누적 집중 시간을 기록한다.
- 마지막 참여자가 나가면 방은 해산된다.
- 방장이 나가고 참여자가 남아 있으면 가장 오래된 참여자에게 방장이 위임된다.
- 방장만 다른 참여자를 강퇴할 수 있다.
- 방장은 자기 자신을 강퇴할 수 없다.

**수용 기준**

- 비공개방 초대 코드가 틀리면 입장할 수 없다.
- 강퇴된 참여자는 방에서 제거된다.
- 퇴장/강퇴 시 진행 중 집중 시간은 정산된다.

### 12-4. 참가자 상태

**기능 요구사항**

- 참여자는 집중 상태로 전환할 수 있다.
- 참여자는 휴식 상태로 전환할 수 있다.
- 이미 집중 중이면 다시 집중 시작할 수 없다.
- 집중 중이 아닌 사용자는 부적절한 휴식 전환을 할 수 없다.
- 종료된 참여자는 상태 변경할 수 없다.

**수용 기준**

- 상태 전환 규칙 위반은 도메인 예외로 실패한다.
- 누적 집중 시간은 FocusTimer 기준으로 계산된다.

### 12-5. 채팅

**기능 요구사항**

- 스터디룸 채팅은 WebSocket으로 제공된다.
- 인증된 사용자만 채팅을 보낼 수 있다.
- 방 참여자만 해당 방의 채팅을 주고받을 수 있다.
- 채팅 메시지는 길이와 공백 검증을 통과해야 한다.

**수용 기준**

- 잘못된 목적지 또는 미참여 사용자의 채팅은 거부된다.
- 유효하지 않은 메시지는 에러 payload로 응답된다.

## 13. Accountability

Accountability는 친구 감시 기능이다. 부모/교사 감시가 아니라 상호 동의한 친구가 제한된 진행 데이터만 볼 수 있다.

### 13-1. 관계 생성

**기능 요구사항**

- 사용자는 본인을 subject로 하는 감시 관계를 생성한다.
- 사용자당 subject 관계는 하나만 가질 수 있다.
- 공개 범위는 `VisibilityPolicy`로 관리한다.
- 관계 생성 시 초대 코드가 생성된다.

**수용 기준**

- 이미 관계가 있으면 중복 생성할 수 없다.
- 공개 범위가 비어 있거나 유효하지 않으면 실패한다.

### 13-2. 초대 코드

**기능 요구사항**

- subject는 감시자가 연결되기 전 초대 코드를 재발급할 수 있다.
- 초대 코드는 만료 시간을 가진다.
- 초대 코드 상태를 조회할 수 있다.

**수용 기준**

- 만료된 초대 코드는 사용할 수 없다.
- 이미 감시자가 연결된 상태에서는 초대 코드 재발급이 제한된다.

### 13-3. 친구 참여/해제

**기능 요구사항**

- watcher는 초대 코드로 관계에 참여한다.
- subject 자신은 자기 관계의 watcher가 될 수 없다.
- subject는 watcher 연결을 끊을 수 있다.
- watcher는 자신이 보고 있는 관계를 끊을 수 있다.

**수용 기준**

- 잘못된 초대 코드는 실패한다.
- 이미 watcher가 있으면 추가 참여할 수 없다.
- 관계 해제 후 watcher와 inviteCode는 제거된다.

### 13-4. 감시자 조회

**기능 요구사항**

- watcher는 허용된 공개 범위에 한해 subject의 요약 데이터를 조회한다.
- 조회 대상은 집중 요약, TopPick 요약, Timetable 요약이다.
- 단일 날짜와 기간 조회를 지원한다.
- 날짜 범위는 검증되어야 한다.

**수용 기준**

- 공개되지 않은 모니터링 대상은 조회할 수 없다.
- watcher가 아닌 사용자는 조회할 수 없다.

## 14. Notification

Notification은 학습 루프를 방해하지 않는 범위에서 복귀와 실행을 돕는다.

### 14-1. 알림 설정

**기능 요구사항**

- 사용자는 알림 종류별 활성화 여부를 설정한다.
- 학교 시간 무음, 주말 학교 시간 무음, 수면 시간 무음, 하루 최대 푸시 수를 관리한다.
- 수면 시간 무음은 해제할 수 없다.
- 학교 시간은 기본 08:00-15:30이다.
- 하루 최대 푸시는 0-10 범위로 제한한다.

**수용 기준**

- 수면 시간 무음을 false로 저장할 수 없다.
- 학교 시간 시작은 종료보다 빨라야 한다.
- 설정이 없으면 기본 설정을 생성해서 반환한다.

### 14-2. 알림 발송 게이트웨이

**기능 요구사항**

- 발송 전 사용자 알림 설정을 확인한다.
- 학교 시간, 수면 시간, 하루 최대 횟수, 타입별 하루 제한을 확인한다.
- 모든 조건을 통과한 경우에만 실제 발송한다.
- 발송 성공 시 delivery log를 저장한다.
- 안전 발송 메서드는 실패를 삼키고 실패 결과를 반환한다.

**수용 기준**

- 비활성화된 알림 타입은 발송하지 않는다.
- 학교 시간/수면 시간에는 발송하지 않는다.
- 하루 제한을 초과하면 발송하지 않는다.

### 14-3. Web Push

**기능 요구사항**

- VAPID public key를 조회할 수 있다.
- 사용자는 Web Push subscription을 등록할 수 있다.
- 같은 endpoint는 재등록 시 갱신한다.
- 비활성 구독은 soft revoke 처리한다.

**수용 기준**

- 잘못된 subscription payload는 실패한다.
- Web Push 비활성 설정에서는 실제 발송하지 않는다.

### 14-4. D-Day / Timing Push

**기능 요구사항**

- D-Day 알림 스케줄러는 설정된 cron에 따라 시험 일정 기반 알림을 보낸다.
- Timing Push는 집중 추천 시간대에 맞춰 알림을 보낸다.
- 발송 여부는 NotificationGateway 정책을 따른다.

**수용 기준**

- 알림 설정이 꺼져 있으면 스케줄러가 실행되어도 발송하지 않는다.
- 학교 시간과 수면 시간 정책을 우회하지 않는다.

## 15. Focus Statistics

통계 기능은 사용자의 집중 패턴을 요약하고 추천한다.

**기능 요구사항**

- 일간, 주간, 월간 집중 통계를 조회한다.
- 시간대별 집중 통계를 조회한다.
- 사용자의 선호 시간과 실제 기록을 바탕으로 집중 추천을 제공한다.
- 진행 중 세션과 완료 세션을 구분해서 계산한다.
- 기간 계산은 서비스 시간대와 일치해야 한다.

**수용 기준**

- 잘못된 기간 범위는 실패한다.
- 데이터가 없는 경우에도 빈 통계 응답이 안정적으로 반환된다.
- 추천 기능은 사용자별 기록과 행동 프로필을 반영한다.

## 16. Home Today

Home Today는 오늘 화면에 필요한 여러 기능의 데이터를 한 번에 조립한다.

**기능 요구사항**

- 오늘 DailyPlan을 조회하거나 생성한다.
- TopPick, Timetable, Focus Sessions, Recovery Card를 포함한다.
- Future Vision, Behavior Profile, Next Exam Schedule은 없으면 null로 반환한다.
- 알림 설정과 친구 감시 상태를 포함한다.
- 활성 Focus Session을 별도 필드로 제공한다.
- 가입 후 7일이 지나면 집중 시간대 카드 노출 여부를 true로 반환할 수 있다.

**수용 기준**

- 선택 설정이 없어도 홈 조회는 실패하지 않는다.
- 필수 데이터인 오늘 DailyPlan과 Focus Sessions는 정상 조립된다.
- 진행 중 세션이 있으면 `activeFocusSession`에 포함된다.

**설계 주의**

- 현재 Home Today는 여러 BC의 query service와 repository를 직접 조합한다.
- 기능상 aggregator 역할이지만 결합도가 높으므로 장기적으로 홈 전용 read model 또는 query port 분리를 검토한다.

## 17. Analytics

Analytics는 제품 가설 검증과 홈/회복 카드 조건 계산에 사용된다.

### 17-1. 이벤트 기록

**기능 요구사항**

- 인증 사용자는 분석 이벤트를 기록할 수 있다.
- 이벤트 타입과 property를 저장한다.
- property key/value는 길이와 null 검증을 통과해야 한다.
- 주요 서비스는 내부적으로 이벤트를 안전 기록할 수 있다.

**수용 기준**

- 잘못된 이벤트 타입 또는 property는 실패한다.
- 안전 기록 실패는 본 기능 성공을 막지 않는다.

### 17-2. Activation Funnel

**기능 요구사항**

- `SIGNUP`, `ONBOARDING_STARTED`, `BEHAVIOR_PROFILE_CREATED`, `ONBOARDING_SKIPPED`, `FOCUS_SESSION_STARTED`, `FOCUS_SESSION_COMPLETED` 이벤트를 퍼널에 투영한다.
- 각 퍼널 단계는 최초 시점만 기록한다.
- 가입 출처와 학년 정보는 가능한 경우 저장한다.

**수용 기준**

- 지원하지 않는 이벤트는 퍼널에 영향을 주지 않는다.
- 같은 단계 이벤트가 반복되어도 최초 값이 유지된다.

## 18. 파일/이미지

**기능 요구사항**

- 프로필 이미지는 외부 스토리지에 업로드한다.
- 허용 확장자와 빈 파일 여부를 검증한다.
- 교체 또는 실패 시 기존/신규 파일 정리 로직을 제공한다.

**수용 기준**

- 빈 multipart file은 저장할 수 없다.
- 지원하지 않는 확장자는 실패한다.
- 회원가입 실패 시 업로드된 프로필 이미지가 정리된다.

## 19. 미구현 또는 후순위 기능

### 19-1. AI 성장 리포트

PRD의 유료 상품인 AI 성장 리포트는 현재 백엔드 구현 범위에 포함되어 있지 않다.

**예상 요구사항**

- 최소 30일 이상의 행동 데이터 수집
- 사용자의 집중, 실패, 회복, TopPick, 회고 패턴 분석
- AI 리포트 생성 요청/상태/결과 저장
- 결제 상태와 리포트 생성 권한 연결
- 학생 직접 결제 및 환불 정책 반영

### 19-2. 프론트엔드/PWA

현재 저장소는 백엔드 중심이며, PWA 설치, 오프라인, 모바일 UI 구현은 별도 범위다.

## 20. 비기능 요구사항

### 20-1. 보안

- 인증이 필요한 모든 API는 JWT를 요구한다.
- CORS는 credentialed request에서 wildcard origin을 허용하지 않는다.
- Refresh Token은 해시해서 저장한다.
- WebSocket 채팅도 인증과 참여자 검증을 수행한다.

### 20-2. 데이터 정합성

- 사용자별 unique 제약은 DB와 애플리케이션 양쪽에서 방어한다.
- 중복 생성 가능성이 있는 기능은 DB 중복 예외를 처리한다.
- Aggregate 내부 불변식은 도메인 메서드에서 검증한다.

### 20-3. 테스트

- 애플리케이션 서비스는 Mockito 단위 테스트를 우선한다.
- JPA 매핑이 중요한 요약 저장 기능은 `@DataJpaTest`를 사용한다.
- 컨트롤러는 MockMvc 기반 테스트를 사용한다.
- 주요 기능 변경 시 서비스 테스트와 컨트롤러 테스트를 함께 갱신한다.

### 20-4. 운영 설정

- 운영 DB는 MySQL을 기준으로 한다.
- 테스트 DB는 H2 MySQL mode를 사용한다.
- Web Push, FCM 등 외부 발송 기능은 enable flag로 제어한다.
- 기본 시간대는 `Asia/Seoul`이다.
