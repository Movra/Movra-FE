# Frontend Conventions

이 문서는 Movrara 프론트엔드 Agent가 따라야 할 프론트엔드 개발 규칙이다. 특정 프레임워크가 확정되기 전까지는 공통 원칙을 우선하고, 스택이 확정되면 해당 섹션을 구체화한다.

## 1. 기본 원칙

- 현재 작업의 PRD/API/기능명세 범위만 구현한다.
- 사용자가 관찰할 수 있는 동작을 기준으로 컴포넌트와 테스트를 설계한다.
- UI, 상태, API, 비즈니스 규칙의 책임을 섞지 않는다.
- 단일 화면에서만 쓰는 로직은 먼저 가까운 곳에 둔다.
- 같은 패턴이 3회 이상 반복되거나 책임이 명확해질 때만 공용화한다.
- 명세에 없는 UX, 필드, 상태, 자동 보정 로직은 만들지 않는다.

## 2. 컴포넌트 작성 원칙

컴포넌트는 하나의 명확한 책임을 가진다.

- 화면 컴포넌트: 라우트 단위 조합, 데이터 요청 연결, 페이지 상태 조정
- 섹션 컴포넌트: 화면 안의 독립적인 UI 영역
- 입력 컴포넌트: label, value, error, disabled, required 등 폼 책임
- 표시 컴포넌트: props를 받아 렌더링만 수행

규칙:

- 컴포넌트 이름은 사용자의 도메인 언어를 우선한다.
- props는 실제 필요한 값만 받는다.
- boolean props가 3개 이상 늘어나면 책임 분리를 검토한다.
- children/render prop/generic component는 반복 근거가 있을 때만 사용한다.
- 컴포넌트 내부에서 API response 원본을 깊게 탐색하지 않는다. 필요한 UI model로 변환한다.
- 긴 파일은 먼저 책임을 확인한다. 무조건 분리하지 않는다.

## 3. 상태 관리 원칙

상태는 가장 좁은 소유 범위에 둔다.

- 입력 중인 값, 토글, 모달 열림 여부: local state
- URL로 공유되어야 하는 필터, 탭, 상세 id: route/query params
- API에서 온 데이터: server state
- 여러 독립 화면에서 공유되는 사용자/설정: global state 후보
- 계산 가능한 값: state로 저장하지 말고 파생한다.

금지:

- 한 화면에서만 쓰는 값을 전역 store에 넣기
- API cache와 client global state를 중복 저장하기
- 서버 응답을 임의로 변형해 원본 계약을 잃어버리기
- optimistic update를 명세/검증 없이 추가하기

## 4. API 호출 계층 원칙

API 연동은 `docs/API-specification.md`를 기준으로 작성한다.

- method, path, request field, response field, status, error code를 명세와 맞춘다.
- endpoint별 함수는 UI 컴포넌트와 분리한다.
- 인증 토큰, base URL, 공통 에러 파싱은 한 곳에서 처리한다.
- API response 타입은 사용하는 필드만 모델링하되 명세 이름을 바꾸지 않는다.
- request body에 명세 없는 필드를 넣지 않는다.
- 날짜/시간 형식은 API 명세의 포맷을 그대로 따른다.
- multipart/form-data, JSON, query param 구분을 명세와 맞춘다.

에러 처리:

- 네트워크 실패, 인증 실패, validation 실패, 서버 오류를 구분한다.
- 사용자에게 보여줄 메시지는 기능명세 또는 제품 문구가 있을 때 따른다.
- 명세에 없는 에러 문구를 새로 만들면 완료 보고에 표시한다.

## 5. UI와 비즈니스 로직 분리

- 컴포넌트는 렌더링과 사용자 이벤트 연결에 집중한다.
- 도메인 규칙, request 생성, response 변환은 별도 함수로 분리한다.
- 단순한 1회성 변환은 가까운 파일에 둔다.
- 공유 유틸은 서로 다른 기능에서 실제로 재사용될 때만 만든다.
- 비즈니스 규칙 함수는 가능하면 순수 함수로 작성하고 unit test를 붙인다.

## 6. 에러/로딩/빈 상태 처리

API 데이터를 표시하는 UI는 다음 상태를 명시한다.

- loading: 사용자가 대기 중임을 알 수 있어야 한다.
- error: 재시도 가능 여부와 다음 행동이 보여야 한다.
- empty: 데이터가 없을 때의 정상 상태를 보여야 한다.
- success: 명세의 핵심 정보가 빠짐없이 보여야 한다.

규칙:

- loading 중 레이아웃이 크게 흔들리지 않게 한다.
- error 상태를 콘솔 로그로만 처리하지 않는다.
- empty 상태를 빈 화면으로 방치하지 않는다.
- retry 버튼은 실제 재요청과 연결될 때만 둔다.
- mock 데이터로 empty/error 상태를 숨기지 않는다.

## 7. 접근성 기본 규칙

기본 접근성은 완료 기준에 포함한다.

- 버튼은 실제 `button` 역할을 사용한다.
- 링크 이동은 link 역할을 사용한다.
- 폼 입력에는 label 또는 접근 가능한 이름이 있어야 한다.
- 에러 메시지는 관련 입력과 연결한다.
- 키보드만으로 주요 흐름을 사용할 수 있어야 한다.
- focus 상태를 제거하지 않는다.
- 모달/다이얼로그는 focus 이동과 닫기 동작을 제공한다.
- 이미지에는 의미 있는 대체 텍스트를 제공한다. 장식 이미지는 숨긴다.
- 색상만으로 상태를 전달하지 않는다.

## 8. 라우팅 규칙

- 라우트는 PRD/기능명세의 사용자 흐름을 기준으로 만든다.
- 인증 필요 여부를 라우트 수준에서 명확히 한다.
- URL에 들어가야 하는 상태와 local state를 구분한다.
- 중첩 라우트는 화면 구조가 실제로 중첩될 때만 사용한다.
- 없는 경로, 권한 없음, 로딩, 에러 상태를 고려한다.
- PRD에 없는 deep link나 redirect 정책은 만들지 않는다.

## 9. 스타일링 규칙

스택 확정 전까지 다음을 따른다.

- 전역 스타일은 reset, theme token, app shell 수준으로 제한한다.
- 컴포넌트별 스타일은 컴포넌트와 가까운 곳에 둔다.
- 색상, 간격, radius, shadow는 반복되는 토큰으로 관리한다.
- 카드 안에 카드를 중첩하지 않는다.
- 텍스트가 버튼/카드/패널 밖으로 넘치지 않게 한다.
- 반응형 레이아웃은 고정 폭보다 `min`, `max`, `grid`, `flex`, `aspect-ratio`를 우선한다.
- 의미 없는 장식 요소를 추가하지 않는다.
- 디자인 시스템이 확정되기 전에는 공용 컴포넌트 수를 최소화한다.

## 10. 테스트 전략

테스트는 완료 판단의 핵심 증거다.

우선순위:

1. 도메인 규칙/변환 함수 unit test
2. 컴포넌트 상태와 사용자 상호작용 test
3. API 경계 통합 test
4. 핵심 사용자 흐름 e2e smoke test

규칙:

- 가능한 경우 실패 테스트를 먼저 작성한다.
- 테스트 이름은 사용자 행동과 기대 결과를 설명한다.
- 구현 세부 class 이름보다 role, label, text, state를 기준으로 검증한다.
- API mock은 계약을 검증하기 위해 사용한다. 실제 연동을 대체하지 않는다.
- snapshot은 보조 수단이다. snapshot만으로 완료하지 않는다.
- 테스트 삭제/skip은 금지한다. 불가피하면 이유와 대체 검증을 보고하고 사용자 확인을 받는다.
- 접근성 회귀가 쉬운 컴포넌트는 role/name 기반 검증을 포함한다.

## 11. React 프로젝트라면 구체화할 기준

React가 선택되면 다음 기준을 추천한다.

- 언어: TypeScript
- 라우팅: React Router 또는 TanStack Router 중 하나를 프로젝트 초기에 결정
- 서버 상태: TanStack Query
- 폼: 단순 폼은 controlled/native, 복잡한 폼은 React Hook Form
- validation: Zod 또는 Valibot
- 테스트: Vitest + Testing Library + MSW, 핵심 흐름은 Playwright
- 스타일: CSS Modules, Tailwind, 또는 디자인 시스템 결정에 맞춘 단일 방식

React 작성 규칙:

- hook은 `use` 접두사를 사용하고 UI 렌더링과 분리 가능한 상태/효과만 포함한다.
- `useEffect`는 외부 시스템 동기화에만 사용한다. 계산을 위해 사용하지 않는다.
- props drilling이 깊어지기 전까지 Context/store를 도입하지 않는다.
- Context는 자주 바뀌는 큰 객체를 그대로 넣지 않는다.
- TanStack Query 사용 시 query key는 endpoint와 parameter를 안정적으로 표현한다.
- mutation 후 cache invalidate/update 정책은 명세와 사용자 흐름에 맞춘다.
- Testing Library에서는 사용자 관점의 query를 우선한다.

## 12. 프론트엔드 완료 체크리스트

- [ ] 현재 작업 범위가 PRD/기능명세/API에서 확인된다.
- [ ] 화면의 loading/error/empty/success 상태가 정의됐다.
- [ ] API field와 type이 명세와 일치한다.
- [ ] UI와 비즈니스/API 로직이 과도하게 섞이지 않았다.
- [ ] 접근성 기본이 확인됐다.
- [ ] 테스트 또는 대체 검증이 있다.
- [ ] 타입체크와 빌드 결과를 확인했다.
- [ ] 요청되지 않은 기능을 추가하지 않았다.
- [ ] 구조 변경과 동작 변경을 분리했다.
